# ТЗ: Полный редизайн рас Chaos Addon (NeoOrigins, NeoForge 1.21.1)

**Репозиторий:** `github.com/Bucket-yeah/Archive-Builder.git`
**Версия:** 1.0
**Адресат документа:** AI-агент, который будет вносить изменения в код мода.
**Авторы аудита:** код-аудит проведён прямым перекрёстным сопоставлением: список способностей в `origins/*.json` ↔ реально существующие файлы в `powers/<origin>/*.json` ↔ строковые литералы в Java-коде ↔ зарегистрированные Brigadier-команды в `init/ModCommands.java`.

---

## 0. Как пользоваться этим документом

1. Документ построен от общего к частному: сначала — что сейчас сломано и почему (раздел 1), затем — рамка/принципы, по которым чинится и переосмысляется баланс (раздел 2), затем — полная спецификация по каждой из 23 рас (раздел 5, самый объёмный и самый важный раздел), и в конце — чек-лист приёмки (раздел 7).
2. Каждая раса в разделе 5 имеет блок **"Задачи агенту"** — это явный список файлов, которые нужно создать/изменить/удалить, и конкретных багов, которые нужно исправить. Начинай работу именно с этих блоков.
3. Работай по фазам из раздела 4 (P0 → P1 → P2 → P3), а не по порядку рас в документе — это минимизирует риск регрессий.
4. **Критическое правило, действующее для всего документа:** ни одна способность не может быть реализована как "декоративный" `neoorigins:attribute_modifier` с `"amount": 0.0` БЕЗ соответствующей Java-проверки `OriginHelper.hasPower(player, "ровно_такой_же_id_строкой")`. Это правило — причина, по которой сейчас в моде 31 способность из 182 не делает вообще ничего (см. 1.3). Перед тем как пометить задачу выполненной, агент обязан грепнуть кодовую базу на точное вхождение ID способности.
5. После каждой фазы — прогонять методологию самопроверки из раздела 7.2 (три скрипта сверки ID), чтобы не плодить новые версии тех же трёх багов.

## 1. Аудит текущего состояния

### 1.1 Что уже работает хорошо — не трогать без необходимости

Архитектура мода в целом сделана грамотно и не нуждается в переписывании с нуля:

- **Конфиг.** `config/ChaosAddonConfig.java` (256 строк) — все балансные числа вынесены в TOML/AutoConfig и настраиваются за пределами кода (например `eaterChaoticAuraInterval`, `swarmSpawnInterval`). Новые числа для редизайна тоже нужно заводить через этот файл, а не хардкодить.
- **`GeneralPowerHandler.java`** (741 строка) — общий обработчик-катчол, который уже неплохо покрывает пассивки многих рас: сейсмочувствие Ancient Sentinel, каменные стаки, кристальный панцирь, иллюзорная плоть (уклонение + спавн обманки) Nightmare Mimic, дежавю Time Wanderer, цепь контратак Dimension Judge, обратный отсчёт аннигиляции, тикер деспавна и т.д. Это рабочий, не декоративный код — с частицами, звуками и персистентным NBT-состоянием.
- **Активные способности (72 штуки, тип `neoorigins:active_ability`).** Проверены ВСЕ — каждая корректно маппится на зарегистрированную команду в `init/ModCommands.java` через цепочку keybind → JSON → `execute_command` → Brigadier-команда. **Этот слой полностью рабочий, без единого разрыва.** Не путать активки с пассивками — именно пассивки дают почти все 31 "мёртвых" способности (раздел 1.3).
- **Расы с выделенными Handler-классами, проверены построчно и работают:**
  - `Necrovore` — 349 строк, рабочая экономика душ (NBT-счётчик `SOUL_TAG`/`chaos_soul_count`).
  - `BloodSmith` — 215 строк, рабочая экономика "зарядов крови" (1 HP полученного урона = 2 заряда, кап 100, индикация в actionbar).
  - `NeuralHijacker` — 195 строк.
  - `StarOracle` — 243 строки, полностью рабочие `star_foresight`/`cosmic_body`/`star_aura`/`melee_vulnerability`.
  - `MirrorPhantom` — 245 строк.
  - `ChaosEngineer` — 281 строка, включая доказанно рабочий паттерн `no_potions` (отмена эффекта зелья по событию питья) — это образцовый паттерн, который нужно скопировать для починки аналогичных "мёртвых" дебаффов у других рас.
  - `InfernalShepherd` — 231 строка, включая реальный `neoorigins:prevent_action` (огнеиммунность).
  - `ParasiteHandler` — обслуживает `parasitic_mind`, реально проверяет `chaos_addon:parasitic_mind/infection`, механика заражения/свечения/следования за носителем — рабочая и интересная.

### 1.2 Подтверждённые баги (ID-рассинхрон → мёртвый код)

Это не "недоделано", а конкретные опечатки/рассинхроны имени способности между JSON-декларацией и Java-обращением. Код написан, но никогда не выполняется, потому что `OriginHelper.hasPower()` ищет строку, которой не существует.

**Баг №1 — Dimension Judge, "Последний вердикт".**
`GeneralPowerHandler.java:173` ищет `chaos_addon:dimension_judge/balance_of_powers` (множественное число), а реально объявленная и существующая способность называется `balance_of_power` (единственное число, см. `origins/dimension_judge.json` и `powers/dimension_judge/balance_of_power.json`). Из-за этого вся механика автоматического срабатывания "Последнего вердикта" при HP < 20% никогда не запускается.
Дополнительная проблема: текст `description` в самом `balance_of_power.json` описывает **другую** механику (отмена атаки при равном HP), не ту, что реализует осиротевший Java-код (авто-вызов на низком HP). То есть тут наложены два разных сломанных замысла — нужно выбрать один при редизайне (см. спецификацию расы в 5).

**Баг №2 — Deep Navigator, "Портальное зрение".**
`events/DeepNavigatorHandler.java:118` и `:133` ищут `chaos_addon:deep_navigator/portal_vision`, а реально объявленная способность называется `dimensional_sight` (см. `origins/deep_navigator.json`). Механика "подсветка ближайших сущностей на 5 секунд после телепортации" полностью написана, но никогда не включается. Это самый дешёвый фикс в документе — высокая ценность при минимальных трудозатратах (см. P0 в разделе 4).

**Баг №3 — Wandering Gardener, "Язык цветов".**
`events/WanderingGardenerHandler.java:192` и `:222` ищут `chaos_addon:wandering_gardener/flower_tongue`, а реально объявленная способность называется `language_of_flowers`. Из-за этого мертва вся механика: авто-удобрение ближайших культур костной мукой + расстановка "цветочных ловушек", срабатывающих на врагах.
Дополнительно: текст `description` в `language_of_flowers.json` обещает третью, никогда не реализованную механику ("растения сообщают через HUD, что им нужна вода/свет") — при фиксе нужно либо реализовать её отдельно, либо переписать описание под реально существующее поведение (авто-удобрение + ловушки).

**Баг №4 — Time Wanderer, осиротевшие JSON-файлы.**
На диске существуют `powers/time_wanderer/stasis_field.json` и `powers/time_wanderer/time_loop.json`, но они не упомянуты в списке способностей `origins/time_wanderer.json` — то есть раса их физически не может получить. Судя по всему, это черновики, частично замещённые объединённой логикой `temporal_dominion`/`temporal_echo`. Просто удалить нельзя: в `GeneralPowerHandler` уже существует и работает система тегов `chaos_stasis`, которая, похоже, ссылается на замысел этих файлов — нужна реконсиляция (см. 5, Time Wanderer), а не слепое удаление.

**Баг №5 — Necrovore, расхождение лора и реализации (не критично, но указано для полноты).**
`powers/necrovore/soul_drop.json` в описании утверждает, что с убитого существа физически выпадает предмет "Душа Существа", но по факту `NecrovoreHandler.java` ведёт души как абстрактный NBT-счётчик — в моде вообще нет ни одного зарегистрированного кастомного предмета (`init/` не содержит item-реестра). Текст описания вводит игрока в заблуждение. Фикс — косметический (переписать текст), либо концептуальный (см. опционально в 5, Necrovore — миграция на нативный `neoorigins:resource` с HUD-полоской, что отчасти решит и эту проблему).

### 1.4 Проблемы дизайна и баланса

Помимо чисто технических багов, у текущего набора рас есть смысловые проблемы:

- **Тематические дубли.** Несколько пар рас целятся в одну и ту же фантазию игрока, из-за чего одна раса в паре ощущается как "урезанная копия" другой. Полный разбор и директивы по разведению — в разделе 3.
- **Отсутствие визуальной идентичности.** Ни одна из 23 рас не имеет постоянного визуального отличия — только временные эффекты свечения. Нативные типы `neoorigins:model_color` / `entity_model` / `shader` / `overlay` не используются вообще нигде в моде (см. 1.5).
- **Разрыв "фантазия vs реализация"** — для рас 🔴-уровня (раздел 1.6) именная, заглавная механика расы физически не существует в коде: Deep Geomancer заявлен как "чувствующий землю", но не имеет ни рудного чутья, ни подземного слуха; Mycelial Symbiont заявлен как "грибная сеть", но его сетевой механики (`moss_network`) не существует; Dimension Judge заявлен как "всевидящий судья", но `all_seeing_eye` не работает.

### 1.5 Каталог нативных типов способностей NeoOrigins (неиспользуемый потенциал)

Извлечён `neoorigins-2.2.5+1.21.1` jar в `/tmp/neoorigins_extract`. Файл `data/neoorigins/schema/power.schema.json` описывает **150+** нативных типов способностей, из которых мод сейчас использует ничтожную долю: 75× `attribute_modifier`, 72× `active_ability`, 26× `multiple`, 7× `condition_passive`, и по 1× `prevent_action`/`effect`/`action_on_event`/`persistent_effect`. В том же jar лежит библиотека готовых референсных origin'ов (`data/neoorigins/origins/powers/*.json` — necromancer, water_mage, fire_mage, golem, kraken, draconic, automaton_ascended и др.), которые можно использовать как живые примеры. Перечень высокоценных неиспользуемых типов:

- **`neoorigins:resource`** — нативный ресурс-бар с HUD-рендером (`hud_render: {label, color}`, `min/max/start_value/regen_rate/regen_interval`). Референс: `necromancer_essence.json`. **Рекомендация:** перевести самодельные NBT-счётчики (заряды крови Blood Smith, души Necrovore, феромоны Swarm Lord, энергию хаоса Eater of Worlds) на этот нативный тип — меньше кастомного кода, готовый HUD из коробки.
- **`neoorigins:summon_minion`** — нативный призыв питомца с кулдауном/деспавном/экипировкой/стоимостью ресурса. Референс: `necromancer_summon_skeleton.json`. Может упростить кастомные команды призыва.
- **`neoorigins:low_hp_threshold`** — декларативный триггер "при HP ниже X% — применить эффекты", без поллинга тиков. Референс: `automaton_ascended_overclock.json`. **Прямое и чистое решение для Dimension Judge** — заменяет собой сломанный тик-поллинг "Последнего вердикта" (баг №1).
- **`neoorigins:dodge_chance`**, **`neoorigins:enhanced_vision`** (параметр `exposure` — это буст яркости/видимости, а **не** буквальная подсветка блоков сквозь стены — важный нюанс для редизайна Geomancer), **`neoorigins:ground_slam`**, **`neoorigins:repulse`**, **`neoorigins:shadow_orb`** (автономные орбитальные сферы), **`neoorigins:scare_entities`** (список типов сущностей), **`neoorigins:item_magnetism`**, **`neoorigins:phantom_form`** (тоггл невидимость+антигравитация), **`neoorigins:thorns_aura`/`thorns_on_hit`**, **`neoorigins:biome_buff`/`damage_in_biome`** (нативно подходит Biomorph), **`neoorigins:modify_lava_speed`/`lava_vision`** (нативно подходит Infernal Shepherd), **`neoorigins:become_dragon`** (идея дикой ультимативной способности), **`neoorigins:model_color`/`entity_model`/`shader`/`overlay`** (сейчас не используются нигде — главный незадействованный потенциал визуальной идентичности).

## 2. Принципы редизайна (рамка для всех 23 рас)

Чтобы редизайн не превратился в очередной набор красивых описаний поверх нулевых маркеров, каждая раса собирается по единому каркасу — **"кит расы"**:

1. **2 пассивки идентичности** (always-on) — то, что игрок чувствует постоянно, не нажимая кнопок. Должны быть тематически считываемы за 5 секунд по названию.
2. **1 условная/сигнатурная пассивка** — срабатывает по триггеру (низкое HP, ночь, биом, серия ударов и т.п.) — именно она формирует "фантазийный момент" расы.
3. **2 активные способности** (первичная и вторичная, на разные клавиши) — с явным ресурсом/стоимостью и кулдауном. Здесь мод уже силён технически (раздел 1.1) — основная работа тут декоративная/балансная, не структурная.
4. **1 ультимативная способность** — долгий кулдаун (≥3–5 минут), мощный, запоминающийся эффект, желательно с эффектным визуалом (частицы/звук/временная трансформация).
5. **2 драбэка** — один "мягкий" (декларативный JSON-нативный, например `damage_in_biome`, `prevent_action`), один "жёсткий" (требует Java-обработки события). У обоих должна быть проверяемая реализация.

**Definition of Done для любой способности в этом документе:** способность не может считаться готовой, если:
- (а) это `attribute_modifier` с `amount: 0.0`/нулевым эффектом **без** парной Java-проверки `OriginHelper.hasPower(player, "<точно совпадающий ID>")`, или
- (б) Java-код существует, но обращается к ID, который не совпадает дословно со строкой в `origins/<race>.json` и `powers/<race>/<power>.json` (это и есть баги №1–3 из раздела 1.2 — механизм их появления одинаковый, и новые правки обязаны на него не наступать).

Перед закрытием задачи по способности агент обязан выполнить пункт чек-листа 7.2 (грep на точное совпадение ID) для каждой новой/изменённой способности.

## 3. Карта тематических пересечений и директивы по разведению

Четыре пары рас сейчас целятся в одну фантазию игрока, и одна раса в каждой паре ощущается урезанной копией другой. Ниже — явное разведение ролей, обязательное к соблюдению при заполнении раздела 5.

**(A) Parasitic Mind ↔ Neural Hijacker — "контроль над чужим телом".**
- Parasitic Mind = **кукловод многих слабых**. Фантазия: расходуемая армия заражённых носителей, играешь от количества и сети, теряешь конкретного носителя — не страшно.
- Neural Hijacker = **похититель одного сильного тела**. Фантазия: ты буквально живёшь внутри чужого, мощного существа, потеря носителя — это полноценная "смерть" билда, требующая поиска нового тела.
Разведение закрепляется числами: у Parasitic Mind много слабых заражённых (5+ одновременно) с дешёвым повторным заражением; у Neural Hijacker — один носитель за раз, долгий кулдаун на смену тела, но носитель наследует значимую часть силы цели.

**(B) Nightmare Mimic ↔ Mirror Phantom — "ты не то, чем кажешься".**
- Nightmare Mimic = **обман/иллюзия**: ложные предметы, обманки-декои, фальшивый лут — пугает и путает противника отсутствием правды.
- Mirror Phantom = **отражение/копирование**: ты честно становишься чужой копией или отражаешь чужой эффект обратно — никакого обмана, чистая симметрия.
Разведение: Nightmare Mimic не должен копировать чужие статы (это территория Mirror Phantom); Mirror Phantom не должен генерировать "фальшивые" объекты (это территория Nightmare Mimic).

**(C) Mycelial Symbiont ↔ Wandering Gardener — "жизнь природы".**
- Mycelial Symbiont = **паразитический контроль территории**: грибница растёт за счёт окружения, агрессивный area-control, узлы сети дают игроку силу за счёт "заражения" местности.
- Wandering Gardener = **мирная поддержка**: пацифистский/около-пацифистский билд (уже подтверждено: `peaceful_soul.json` — прямой урон 0 как нативный атрибут, альтернативный бой через приручение), помогает другим/себе расти, не паразитирует.
Разведение: у Symbiont механики должны явно отбирать/конвертировать ресурсы окружения; у Gardener — выращивать/делиться, не отбирая.

**(D) Time Wanderer ↔ Star Oracle ↔ Deep Navigator — "не совсем обычное перемещение/информация".**
- Time Wanderer = **гэмблер временного хаоса**: рандом, дежавю, нестабильность — ты не контролируешь исход, только повышаешь шансы.
- Star Oracle = **информационный glass cannon на дистанции**: предвидение, хрупкость (cosmic_body уже это подтверждает — нет иммунитета к голоду/яду/wither + повышенный урон), сила в информации и дистанционном давлении.
- Deep Navigator = **специалист мобильности/блинков**: чистое контролируемое перемещение в пространстве (телепорты, пространственная память), без рандома Time Wanderer и без хрупкости/дальнобойности Star Oracle.

## 7. Чек-лист приёмки и методология самопроверки

### 7.1 Чек-лист приёмки по каждой расе

- [ ] Все пассивки расы либо (а) реальный нативный тип с проверяемым эффектом, либо (б) `attribute_modifier` с `amount: 0.0`, у которого есть парная Java-проверка `OriginHelper.hasPower()` с ID, совпадающим **дословно**.
- [ ] Все ID способностей, используемые в Java (`OriginHelper.hasPower("chaos_addon:<race>/<power>")`), совпадают дословно с ID, объявленными в `origins/<race>.json` и именем файла `powers/<race>/<power>.json`.
- [ ] Все JSON-файлы в `powers/<race>/` упомянуты в списке способностей `origins/<race>.json` (нет осиротевших файлов).
- [ ] Текст `description` в каждом power-JSON соответствует реальному поведению, реализованному в Java (а не альтернативной/устаревшей версии замысла).
- [ ] Все новые активные способности зарегистрированы как Brigadier-команды в `init/ModCommands.java` и привязаны через `execute_command`.
- [ ] Все новые балансные числа (кулдауны, интервалы, проценты, лимиты) вынесены в `config/ChaosAddonConfig.java`, не захардкожены.
- [ ] Раса не нарушает ни одну из 4 директив разведения тематических пар (раздел 3).

### 7.2 Методология самопроверки (переиспользовать скрипты аудита этой сессии)

Рекомендуется воссоздать и прогонять после каждой фазы (P0–P3) три проверочных Python-скрипта, аналогичных тем, что использовались при первичном аудите:

1. **Сверка способностей origin ↔ файлы:** для каждого `origins/<race>.json` распарсить список ID способностей и сравнить 1:1 со списком файлов в `powers/<race>/` — найти (а) ID без файла, (б) файлы без ID в списке (осиротевшие).
2. **Сверка нулевых маркеров ↔ Java-литералы:** найти все power-JSON с паттерном `attribute_modifier`+`amount: 0.0`, для каждого ID сделать `grep -r "chaos_addon:<race>/<power>"` по всему Java-исходнику — если совпадений ноль, способность декоративна (баг того же типа, что 31 найденная в этой сессии).
3. **Сверка `execute_command` ↔ зарегистрированные команды:** для каждого power-JSON с `type: neoorigins:active_ability` и `execute_command` сверить указанную команду со списком литералов, зарегистрированных в `init/ModCommands.java`.

Любое расхождение, найденное этими тремя скриптами после внесения изменений — это регрессия, повторяющая один из багов №1–3 из раздела 1.2, и должна быть исправлена до закрытия задачи.
